# Диагностика инцидента: задержка команды «В главное меню» из-за активности другого пользователя

Дата диагностики: 2026-02-14 (Екатеринбург, UTC+5)

## 1. Симптом
Пользователь A отправил команду `В главное меню` около `20:59` и получил ответ с заметной задержкой.
При этом в этот же момент пользователь B выполнял генерацию изображения.

Ключевая особенность инцидента: команда навигации пользователя A зависела от длительной операции пользователя B (межпользовательская блокировка).

## 2. Подтвержденные факты из логов
По логам Modal (`modal app logs nano-banana-bot --timestamps`) за 2026-02-14:

- `20:58:59+05:00` начат долгий вызов генерации (`GenAI request generate_final_image`).
- `20:59:31+05:00` получен результат генерации (`Generated image: ...`).
- `20:59:33+05:00` завершен HTTP webhook-запрос длительностью `45.1 s`:
  `POST / -> 200 OK (duration: 45.1 s, execution: 45.0 s)`.
- Сразу после этого, в `20:59:34+05:00`, отработали короткие запросы:
  - `POST / -> 200 OK (duration: 702.2 ms, execution: 611.7 ms)`
  - `POST / -> 200 OK (duration: 1.23 s, execution: 622.2 ms)`

Интерпретация:
- Пока выполнялась генерация (~45 секунд), другие апдейты не получили быстрого CPU-времени в том же инстансе.
- Команда `В главное меню` обработалась после освобождения обработчика долгого запроса.

## 3. Диагноз (root cause)
### Основная причина
В асинхронном обработчике используются блокирующие (синхронные) сетевые вызовы, которые выполняются в event loop и блокируют обработку других апдейтов.

Подтверждения в коде:
1. Синхронный вызов GenAI внутри `async` функции:
- `/Users/repnikovv/Проекты dev/nano-banana-bot (Py Modal)/gemini.py:68`
- `/Users/repnikovv/Проекты dev/nano-banana-bot (Py Modal)/gemini.py:144`

Обе строки вызывают `client.models.generate_content(...)` без `await` и без вынесения в отдельный поток/воркер. Это блокирующая операция на десятки секунд.

2. Синхронный HTTP к Telegram внутри `async` функции:
- `/Users/repnikovv/Проекты dev/nano-banana-bot (Py Modal)/bot.py:120`

`requests.get(...)` также блокирует event loop (пусть и на меньший срок).

### Усиливающий фактор
Webhook-обработчик не изолирует «долгие» задачи от «быстрых» команд навигации:
- `/Users/repnikovv/Проекты dev/nano-banana-bot (Py Modal)/main.py:27-29`
- `/Users/repnikovv/Проекты dev/nano-banana-bot (Py Modal)/main.py:48`

`await dp.feed_update(...)` обрабатывает апдейт в рамках того же запроса/контекста. При блокировке event loop latency накапливается для других апдейтов.

## 4. Причинно-следственная цепочка (как именно это происходит)
1. Пользователь B запускает генерацию (долгий путь: загрузка фото -> анализ стиля -> генерация).
2. В `async`-коде вызывается синхронный `generate_content`, который блокирует event loop.
3. В это время приходит апдейт пользователя A с `В главное меню` (это быстрый путь по бизнес-логике).
4. Быстрый апдейт не может мгновенно выполниться, потому что loop занят блокирующей задачей.
5. После завершения генерации долгий webhook закрывается, и только затем обрабатываются накопившиеся короткие апдейты.

## 5. Почему это именно межпользовательская проблема
FSM/сессии в `Storage` раздельные по `user_id`, поэтому логически пользователи независимы.
Но исполнение кода происходит в общем runtime-контексте контейнера/event loop, и блокировка loop одним запросом влияет на всех пользователей, пришедших в этот интервал.

## 6. Влияние на продукт
- Навигационные команды (`В главное меню`, `Отмена`) перестают быть мгновенными под нагрузкой.
- Пользовательский UX деградирует: ощущение «бот завис».
- Рост p95/p99 latency на фоне одновременных генераций.
- Масштаб проблемы растет с числом параллельных пользователей.

## 7. Дополнительные технические наблюдения
1. В `download_file` нет проверки `status_code`, `Content-Type` и таймаута:
- `/Users/repnikovv/Проекты dev/nano-banana-bot (Py Modal)/bot.py:120-121`
Это отдельный риск нестабильности/ошибок входных данных.

2. Для данного инцидента ключевым фактором была именно задержка event loop, а не ошибка валидации параметров модели.

## 8. Что уже улучшено в диагностике
В проект уже добавлены подробные логи входных параметров для вызовов Gemini (`GenAI request ...`) и no-image diagnostics.
Это повышает наблюдаемость причин ошибок генерации, но не устраняет межпользовательскую блокировку по latency.

## 9. Диагностические пробелы (для архитектора/разработчика)
Сейчас в логах нет полного correlation trail на каждый апдейт:
- `update_id`
- `user_id`
- тип команды/состояние FSM на входе
- timestamps начала/конца каждого этапа обработки

Без этого трудно на 100% связать конкретный delayed message с конкретным webhook-запросом в одном отчете (есть сильная косвенная корреляция по времени, но нет явной ID-связки).

## 10. Формулировка проблемы для проектирования решения
Текущая реализация допускает блокировку обработки апдейтов всех пользователей при выполнении длительных синхронных операций в контексте webhook-обработчика. Это приводит к межпользовательской очереди и задержкам даже для быстрых команд управления.

Критерий устранения проблемы (в терминах диагностики, без выбора конкретной архитектуры):
- Команды навигации обрабатываются независимо от длительных генераций других пользователей.
- Под нагрузкой отсутствуют случаи, где короткие команды ждут завершения чужой генерации.
- В логах есть достаточная трассировка, чтобы однозначно доказать отсутствие межпользовательской блокировки.
